# GUARDRAILS.md — 开发行为护栏

> 基于 [`ai-dev-guardrails`](E:\AISkills\Nagi_Skills\ai-dev-guardrails) v3.0 五层防御架构，
> 适配 RoboMind-S3 项目的 20 版本迭代模型。

---

## 核心铁律

> **一旦 Claude Code（架构师）做出可能偏离版本规划的错误决策——无论是技术认知不足、
> 范围蔓延、不可行承诺、还是架构矛盾——任何实现者 Agent 必须立即暂停并提示拉回。**

这不是"不信任"，而是"双层校验"——架构师定方向，护栏保底线。

---

## 五层防御适配

```
Layer 1: 指令层  L0-L5 决策流     — 拦截不合理/模糊/错误需求
Layer 2: 项目层  M1-M5            — 版本契约/范围仪表盘/里程碑关口/规划审计/偏离回拽
Layer 3: 交付层  M6 Reliability   — 每版本交付前回归+迭代健康检查
Layer 4: 审计层  M8 Quality       — 每 5 版本做一次五维代码质量审计
Layer 5: 范围层  M9 Fidelity      — 文件变更审计 + 反过度工程
```

---

## Layer 1: 指令层 — 每轮对话自动触发

适用于本项目的五个决策等级：

| 等级 | 触发条件 | 本项目示例 | 响应 |
|------|----------|-----------|------|
| **L0** | 用户讨论规划/路线图 | "我们要不要加入语音识别？" | 听完→对照 version_roadmap.md 审计→指出矛盾/缺口 |
| **L1** | 轻微模糊 (≥3 决策可推断) | "修一下 Kconfig 默认值" | 陈述假设→开工→标注 |
| **L2** | 明显缺口 (≤2 决策可推断) | "加个摄像头功能" | 停→列出缺失项→定向提问 |
| **L3** | 技术错误 | "ESP32-S3 有硬件 JPEG 编码器" | 停→纠正→提供替代方案 |
| **L4** | 踩红线 | "让 ESP32 跑本地 LLM 训练" | 硬停→解释限制→降级方案 |
| **L5** | 严重项目偏离 | 累积范围 >50% 超出原始契约 | 硬停→展现偏差→重新签约 |

## Layer 2: 项目层 — 版本级监控

### M1: 版本契约 (Per-Version Project Contract)

每个版本启动前，Claude Code 在 version_roadmap.md 中锁定：

```
📋 v0.X.Y 版本契约
────────────────────
范围:   [CX-1, CX-2, CX-3 任务列表]
架构:   [本版本涉及的模块/接口]
排除:   [明确不在本版本做的事]
约束:   [硬件未到/依赖未满足]
里程碑: [验收标准清单]
```

### M2: 范围仪表盘

每完成 3 个版本，检查累积漂移：

| 指标 | 阈值 | 动作 |
|------|------|------|
| 累积新增任务 vs 原始规划 | >30% | 🟡 黄色警报 |
| 新增任务在一轮内未经审查 | >3 个 | 🔴 红色警报 |
| 架构一致性破坏 | 任何一次 | 🟠 橙色警报 |

### M3: 里程碑关口

本项目天然有三个关口（与 20 版本路线图对齐）：

```
🏁 Gate 1: v0.0.5 完成 (架构→编码前)
   所有 5 个骨架版本通过 → 确认可以开始板子启动

🏁 Gate 2: v0.1.9 完成 (核心模块完成)
   板子启动 + 对话打通 + 稳定 → 确认可以扩展摄像头

🏁 Gate 3: v0.1.9→v0.2.0 过渡 (扩展点)
   确认当前功能稳定 → 审核摄像头/TF 设计的架构一致性
```

### M4: 规划审计

Claude Code 在每次规划对话时，后台检查：

- [ ] 新版规划是否与 version_roadmap.md 已确认路线矛盾？
- [ ] 版本顺序是否可行（依赖未满足的版本不能排前面）？
- [ ] 是否假设了尚未确认的硬件能力？
- [ ] 新规划是否增加了超出项目可持续的复杂度？

### M5: 偏离检测与回拽（五步 SOP）

```
步骤1: 检测 → 对比当前方向 vs 版本契约
步骤2: 评级 → Minor (<10%偏移) / Major (20-50%) / Critical (>50%)
步骤3: 拦截 → Major+ 暂停开发
步骤4: 说明 → "v0.X.Y 契约是 [X]，现在在做 [X+Y+Z]"
步骤5: 回拽 → 提供 2-3 修正路径 → 用户选择 → 更新契约
```

**偏离警报模板**:

```
🛡️ 护栏警报：检测到 v0.X.Y 版本偏离

📋 版本契约（已确认）：
   范围：[原始 CX 任务列表]
   架构：[已确认方案]

🔄 当前实际方向：
   已完成：[实际完成]
   新增项：[超出契约的内容]

⚠️ 偏离分析：
   - [具体偏离 1]
   - [具体偏离 2]

建议修正：
🅰️ 缩减到版本契约范围，完成后再扩展
🅱️ 正式更新契约，拆分到新版本
🅲️ 评估新增项的优先级，延后到后续版本

你怎么看？
```

## Layer 3: 交付层 — 每版本必检

### M6: 可靠性关卡（不可绕过）

每个版本交付前，无论实现者是谁（CodeWhale/Reasonix/Codex），Claude Code 必须执行：

```
🔒 回归检查:
- [ ] 本版本修改涉及的所有入口路径已追踪
- [ ] 无副作用影响已稳定的模块
- [ ] 如有测试套件：全部通过
- [ ] 如无测试套件：脑内烟雾追踪（逐路径确认输入→输出不变）
- [ ] 回退路径清晰（可单独撤销本版本变更）

🏥 迭代健康:
- [ ] 代码结构清晰：新代码遵循项目规范（单例/Kconfig/ESP-IDF 日志）
- [ ] 无"下个版本再重构"的累积债
- [ ] 新抽象命名有意义
- [ ] 无死代码/注释掉的实验代码
- [ ] CMakeLists.txt/Kconfig 依赖声明正确
- [ ] 2 周后回头改这版代码不需要交接文档
```

**判定**: PASS+PASS = 可交付。任何 FLAG = 列出问题 + 建议修复时间。

## Layer 4: 审计层 — 里程碑级

### M8: 五维质量审计

每 5 个版本（v0.0.5, v0.1.4, v0.1.9, v0.2.4）执行一次：

| 维度 | ESP32 嵌入项目检查项 |
|------|---------------------|
| ① 架构 | 模块间耦合度、单例模式一致性、FreeRTOS 任务间通信清晰 |
| ② 安全 | API Key 是否硬编码、WiFi 密码存储、TLS 证书验证 |
| ③ 性能 | SPI 频率、DMA 缓冲大小、PSRAM vs SRAM 分配、栈深度 |
| ④ 质量 | 函数长度 <200 行、魔数常量化、无重复代码块 |
| ⑤ 测试 | 关键路径手动验证记录、内存泄漏检查、启动链完整性 |

**严重等级**:
- 🔴 CRITICAL: 阻塞交付 (如 API Key 硬编码)
- 🟠 HIGH: 下版本前修复 (如 PSRAM 未用)
- 🟡 MEDIUM: 方便时修复 (如魔数)
- 🟢 LOW: 跟踪即可 (如命名不一致)

## Layer 5: 范围层 — 文件级

### M9: 范围忠实度四级检测

| 等级 | 触发 | 本项目示例 |
|------|------|-----------|
| **SF-L1** | 1-2 处不必要改动 | 修 display_driver.cpp 时顺手改了 chat_ui.cpp 的注释 |
| **SF-L2** | 创建了未要求的文件/抽象 | 加个 Kconfig 项却新建了整个 helper 类 |
| **SF-L3** | 改了 3+ 个未提及文件 | 修一个 GPIO 默认值→顺手改了 partitions.csv + CMakeLists.txt |
| **SF-L4** | 200+ 行 diff / 级联修复 | 改 SPI 初始化→触发 LVGL 重配→触发 UI 重构 |

**必要性测试** (SF-L2+ 触发): 对每个改动的文件问——"撤销这个文件的改动，需求功能会坏吗？" → 不会 → 立即撤销。

**反过度工程表** (本项目特化):

| Agent 冲动 | 正确做法 |
|-----------|---------|
| "这个函数名不好，改一下" | 不是你的任务。标注但不改。 |
| "加个 try-catch 以防万一" | 这个异常真的会发生吗？不发生就不加。 |
| "提取成工具函数" | 只用了一次？内联比抽象好。 |
| "加个抽象层未来扩展用" | YAGNI。现在需要的才写。 |
| "顺手修一下旁边的 bug" | 一次改一个版本任务。 |
| "把 ESP_LOGI 改成 ESP_LOGW" | 不是你的任务。 |
