## 🔴 CRITICAL

### C1. display_driver.cpp — LVGL Flush CASET/RASET 缺字节

**文件**: `main/display_driver.cpp:358-411` (LvglFlushCallback)

**问题**: CASET 和 RASET 命令的数据传输各缺了 1 个字节。

```cpp
// 当前代码 (有 Bug):
uint8_t caset[4] = {
    0x2A,
    static_cast<uint8_t>((area->x1 >> 8) & 0xFF),  // x1_hi
    static_cast<uint8_t>(area->x1 & 0xFF),           // x1_lo
    static_cast<uint8_t>((area->x2 >> 8) & 0xFF),   // x2_hi
};
uint8_t x2_byte = static_cast<uint8_t>(area->x2 & 0xFF); // ← 声明了但从未发出!

// t2 只发送了 24 bits = 3 bytes (x1_hi, x1_lo, x2_hi)，x2_lo 丢失
t2.length = 24; t2.tx_buffer = &caset[1];
```

**影响**: 对于 240×320 显示屏，x2=239=0x00EF。缺失低字节后显示屏收到 x_end=0x0000=0，列范围只有 1 像素宽。**整个 LVGL 渲染将完全错误。**

RASET (y 方向) 有完全相同的问题。

**修复方案**: 改用单次 5 字节传输发送完整命令：

```cpp
uint8_t caset_cmd[5] = {
    0x2A,
    static_cast<uint8_t>((area->x1 >> 8) & 0xFF),
    static_cast<uint8_t>(area->x1 & 0xFF),
    static_cast<uint8_t>((area->x2 >> 8) & 0xFF),
    static_cast<uint8_t>(area->x2 & 0xFF),
};
```

### C2. wifi_manager.cpp — `esp_event_loop_create_default()` 重复调用

**文件**: `main/wifi_manager.cpp:56`

**问题**: `esp_event_loop_create_default()` 在每次 `Connect()` 中被调用。如 Connect→Disconnect→Connect，第二次调用会失败或泄漏第一个事件循环。

```cpp
bool WifiManager::Connect() {
    // ...
    ESP_ERROR_CHECK(esp_event_loop_create_default());  // ← 每次都创建!
```

**影响**: 
- `ESP_ERROR_CHECK` 在失败时调用 `abort()` → ESP32 重启
- 即使成功也会创建多个事件循环，内存泄漏

**修复方案**: 将 `esp_netif_init()` 和 `esp_event_loop_create_default()` 移到单独的一次性初始化，或在 Connect 中加 `static bool` 守卫。

### C3. wifi_manager.cpp — `ESP_ERROR_CHECK` 导致 abort 而非返回 false

**文件**: `main/wifi_manager.cpp:53-85`

**问题**: 7 处 `ESP_ERROR_CHECK` 在失败时调用 `abort()`，但 `Connect()` 函数签名返回 `bool` 暗示可以优雅地处理错误。

```cpp
ESP_ERROR_CHECK(esp_netif_init());           // abort on fail
ESP_ERROR_CHECK(esp_event_loop_create_default());  // abort on fail
ESP_ERROR_CHECK(esp_wifi_init(&cfg));         // abort on fail
ESP_ERROR_CHECK(esp_wifi_set_mode(...));      // abort on fail
ESP_ERROR_CHECK(esp_wifi_set_config(...));    // abort on fail
ESP_ERROR_CHECK(esp_wifi_start());            // abort on fail
```

**影响**: Wi-Fi 初始化失败时 ESP32 直接重启，无法给用户任何提示。

**修复方案**: 全部替换为 `esp_err_t ret = ...; if (ret != ESP_OK) { ESP_LOGE(TAG, ...); return false; }`

## 🟠 HIGH

### H1. wifi_manager.cpp — `s_retry_count` 跨连接周期泄漏

**文件**: `main/wifi_manager.cpp:32,156-162`

**问题**: `s_retry_count` 是文件级静态变量，只在 `WIFI_EVENT_STA_CONNECTED` 中清零。如果 Connect 超时后再次 Connect（不经过 connected 事件），计数器不会重置。

```cpp
static int s_retry_count = 0;  // ← 文件级静态，Disconnect() 不清零

// HandleWifiEvent:
if (s_retry_count < kMaxRetry) {
    esp_wifi_connect();
    s_retry_count++;  // ← 第二次 Connect 时从上次的值继续累加
```

**修复方案**: 在 `Connect()` 开头加 `s_retry_count = 0;`

### H2. wifi_manager.h/cpp — 双重重试计数器

**文件**: `main/wifi_manager.h:62` + `main/wifi_manager.cpp:32`

**问题**: `.h` 中声明了 `int retry_count_{0}` 成员变量，但 `.cpp` 中实际使用的是文件级 `static int s_retry_count`。成员变量从未被使用（死代码）。

**修复方案**: 删除 `.h` 中的 `retry_count_`，统一使用 `s_retry_count`（或在 Connect 开头重置）。

### H3. wifi_manager.cpp — 事件处理器未注销

**文件**: `main/wifi_manager.cpp:66-71, 113-122`

**问题**: `Connect()` 中注册了事件处理器但 `Disconnect()` 中从未注销。虽然 `esp_event_handler_instance_register` 的最后一个参数 `nullptr` 意味着没有保存句柄用于后续注销，但反复 Connect/Disconnect 会累积注册。

**修复方案**: 保存 `esp_event_handler_instance_t` 句柄并在 `Disconnect()` 中调用 `esp_event_handler_instance_unregister()`。

### H4. wifi_manager.cpp — IP 地址构建使用了非标准宏语法

**文件**: `main/wifi_manager.cpp:184-187`

**问题**: `IP2STR(&evt->ip_info.ip).addr[0]` 语法依赖特定的宏展开实现，在 ESP-IDF 版本间可能变化。

**修复方案**: 使用标准 API `esp_ip4addr_ntoa()`:
```cpp
char ip_str[16];
esp_ip4addr_ntoa(&evt->ip_info.ip, ip_str, sizeof(ip_str));
ip_address_ = ip_str;
```

## 🟡 MEDIUM

### M1. display_driver.cpp — LVGL buffer 偏小

**文件**: `main/display_driver.cpp:126`

**问题**: `buf_size = width * height / 10`。对于 320×480 (ST7796) 仅 15,360 像素 ≈ 30KB。复杂 UI 大面积刷新时会产生撕裂。

**修复**: 修改为 `/ 4` (25% 屏幕缓冲) 或通过 Kconfig 配置 `ROBOMIND_LVGL_BUFFER_FRACTION`。

### M2. display_driver.cpp — DC 引脚在 Flush 回调中重复设置

**文件**: `main/display_driver.cpp:364,394,404`

**问题**: LvglFlushCallback 每次 flush 设置 DC 3 次（CASET 命令、RAMWR 命令、数据），可以用多段 SPI transaction (`spi_device_queue_trans`) 减少事务数。非功能性问题，但影响刷新效率。
