## 🔴 CRITICAL

### C1. main.cpp — ChatEngine::Initialize() 从未调用

**文件**: `main/main.cpp:60-67`

**问题**: ChatEngine 获取了实例、设置了回调，但 **Initialize() 从未被调用**。

```cpp
auto* engine = ChatEngine::GetInstance();
engine->SetMessageCallback(...);
engine->SetStatusCallback(...);
// ❌ engine->Initialize() 缺失!
```

**影响**: `api_endpoint_`/`api_key_`/`model_name_`/`system_prompt_` 全部为空。`SendMessage()` 会用空 URL 发 HTTP 请求 → **所有 AI 对话必然失败**。

**修复**: 设置回调之前调用 `engine->Initialize()`。

### C2. main.cpp — 无输入回调，发送按钮事件链断路

**文件**: `main/main.cpp:57-58`

**问题**: ChatUI 创建后从未调用 `SetInputCallback`。用户输入文字点击发送 → `OnSendClicked` 检查 `input_callback_` → 回调为空 → **什么都不发生**。

```cpp
ui->Initialize();
// ❌ ui->SetInputCallback(...) 缺失!
```

**影响**: 用户在输入框打字点发送 → 文本清空 → 没有任何反应。UI ↔ Engine 之间的唯一桥梁断了。

**修复**: 初始化 UI 后立即绑定输入回调：
```cpp
ui->SetInputCallback([engine](const std::string& text) {
    engine->SendMessage(text);
});
```

### C3. chat_engine.cpp — DoChatRequest 未创建独立 Task，阻塞 LVGL

**文件**: `main/chat_engine.cpp:99-125, 252-336`

**问题**: `.h` 注释说"在独立 task 中执行 HTTP 请求"，但代码直接在主调线程中调用 `DoChatRequest()`。调用链：LVGL timer → UI event → SendMessage → DoChatRequest → esp_http_client_perform (阻塞 30s)。

**影响**:
- SSE 流式期间 **整个 UI 冻结**（LVGL timer handler 被阻塞）
- 长回复可能触发 **看门狗复位**
- ESP32 响应时间 = HTTP 请求耗时

**修复**: 创建独立 FreeRTOS task 执行 HTTP 请求。

## 🟠 HIGH

### H1. main.cpp — Display 初始化失败被静默忽略

**文件**: `main/main.cpp:52-54`

**问题**: Display 初始化失败后只打日志，程序继续执行 LVGL 相关操作。

```cpp
if (!DisplayDriver::GetInstance()->Initialize()) {
    ESP_LOGE(TAG, "Display init failed");
    // ❌ 继续执行 → 创建 UI → 注册 LVGL → 进入 RunLvglLoop → CRASH
}
```

**影响**: Display 失败后，后续 `ChatUI::Initialize()` 创建 LVGL 对象会崩溃（display driver 未就绪）。

**修复**: Display 失败应 `esp_restart()` 或进入安全模式（仅串口）。

### H2. chat_ui.cpp — SSE 流式每 delta 创建一个新气泡

**文件**: `main/chat_ui.cpp:214-224`

**问题**: `AppendMessage` 接收每条 SSE delta 都调用 `AddMessageBubble` 创建新气泡。50 条 delta = 50 个叠加气泡，内存浪费。

```cpp
void ChatUI::AppendMessage(const std::string& role, const std::string& content) {
    if (role == "assistant") {
        pending_assistant_text_ += content;
        AddMessageBubble("assistant", pending_assistant_text_);  // ← 每次新建!
    }
}
```

**修复**: 首次创建时保存容器指针 → 后续 delta 只更新 `content_label` 文本 + 调整高度。

### H3. chat_ui.cpp — `AddMessageBubble` 高度计算依赖未布局的 Label

**文件**: `main/chat_ui.cpp:226-263`

**问题**: `lv_obj_set_height(container, lv_obj_get_height(content_label) + 24)` — `content_label` 刚创建，LVGL 尚未执行布局，`lv_obj_get_height` 可能返回 0。

**影响**: 气泡容器高度为 24px（仅含 padding），内容被裁剪。

**修复**: 改用 `lv_obj_update_layout(content_label)` 强制布局后再取高度。

## 🟡 MEDIUM

### M1. chat_engine.cpp — s_sse_buffer 设为全局静态变量

**文件**: `main/chat_engine.cpp:199-200`

**问题**: `s_sse_buffer` 和 `s_engine_for_http` 是文件级全局静态，非线程安全。虽然 busy_ 守卫防止并发请求，但架构上不干净。

**修复**: 改为 `ChatEngine` 的成员变量。

### M2. main.cpp — NVS ESP_ERROR_CHECK 与 wifi_manager 风格不一致

**文件**: `main/main.cpp:40`

**问题**: v0.0.3 已将 wifi_manager 中所有 `ESP_ERROR_CHECK` 替换为手动错误处理，但 main.cpp 仍使用 `ESP_ERROR_CHECK(ret)` → abort on NVS failure。NVS 失败是致命的，所以 abort 可接受，但风格不一致。

### M3. main.cpp — WiFi 失败直接 restart，无重试计数

**文件**: `main/main.cpp:45-48`

**问题**: WiFi 失败 → `esp_restart()` → 重启后再次失败 → 无限重启循环。应加 NVS 存储的重启计数器。

### M4. chat_engine.cpp — 流结束后历史裁剪代码重复

**文件**: `main/chat_engine.cpp:109-113, 320-324`

**问题**: 同样的裁剪逻辑出现两次（SendMessage 和 DoChatRequest 完成时各一次）。应提取为私有方法 `TrimHistory()`。

## 🟢 LOW

### L1. chat_ui.cpp — ShowSplashScreen 中创建了未使用的 timer_label

**文件**: `main/chat_ui.cpp:80-81`

```cpp
lv_obj_t* timer_label = lv_label_create(screen_);
lv_label_set_text(timer_label, "");
```
创建后从未使用或删除 → 内存泄漏（微小）。

### L2. chat_ui.cpp — AddMessageBubble 无最大气泡数量限制

**文件**: `main/chat_ui.cpp:226`

长对话可能累积数百个 LVGL 对象 → OOM。ChatEngine 有 `max_history_` 但 UI 无对应清理。
